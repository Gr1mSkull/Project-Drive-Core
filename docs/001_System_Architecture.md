# 001 — System Architecture

Версия: **Gen1 v0.1** · Статус: **этап 1 — зафиксировано**

## 1. Обзор

DriveCore Gen1 — платформа из трёх CAN-узлов в автомобиле + внешний клиент UI:

```
АКБ → [Контактор массы] → DCC ──┬── силовые выходы → нагрузки
                                │
                    CAN FD (одна шина, линейная)
                                │
              ┌─────────────────┼─────────────────┐
              │                 │                 │
            ECU            Button Box      (резерв Gen2)
         двигатель         события HMI
              │
         (телеметрия, cooling_request)

DCC ── Wi-Fi/BLE ──► Планшет (Web UI, необязателен)
```

| Узел | Роль | Критичность |
|------|------|-------------|
| **DCC** | PDM + VCM + диагностика + gateway + логи | Критичный |
| **ECU** | Только двигатель | Критичный |
| **Button Box** | События кнопок/энкодеров | Высокий |
| **Планшет** | Отображение Web UI | Низкий (не критичен) |

> **DCC — сервер. Планшет — клиент.** Отказ планшета или ESP32 не останавливает автомобиль.

## 2. Программные слои

| Слой | Платформа | Ответственность |
|------|-----------|-----------------|
| **Real-Time** | STM32 (Logic) | CAN, силовые каналы, VCM, диагностика, SPI |
| **Service** | ESP32 (Radio) | REST, WebSocket, OTA, файлы UI, YAML→binary |
| **Presentation** | Планшет/ПК | Браузер, только отрисовка |

STM32 не парсит JSON и не обслуживает HTTP.

## 3. DCC — внутренняя структура

```
┌─────────────────────────────────────────┐
│         DriveCore Controller (DCC)      │
│  Logic Board   STM32G474, CAN, FRAM     │
│  Power Board   Smart HS, H-bridge, TVS  │
│  Radio Board   ESP32 module (дочерняя)  │
└─────────────────────────────────────────┘
```

| Плата | Меняется в Gen2? |
|-------|------------------|
| Logic | Редко |
| Power | Да (каналы, ключи) |
| Radio | Да (Rev.A→B) |

Внутри корпуса **нет CAN** — только SPI (Logic↔Radio) и шина Logic↔Power.

## 4. CAN Gen1 — архитектурное решение

| Параметр | Значение |
|----------|----------|
| Шин | **1** (CAN FD) |
| Топология | Линейная, без звезды |
| Терминаторы | 120 Ω на концах (DCC + последний узел) |
| Скорость | 500 kbit/s nominal, 2 Mbit/s data (FD) |
| Config master | DCC |
| Протокол | [DriveCore Protocol v0.1](004_Communication_Protocol.md) |

**Почему одна шина:** Gen1 имеет 3 узла; две шины усложняют жгут без выигрыша. Второй CAN-трансивер на Logic Board — резерв / мост на Gen2.

Порядок на шине (пример жгута):

```
DCC ── ECU ── Button Box
 ↑                      ↑
терминатор         терминатор
```

## 5. Электрическая архитектура E30

### 5.1 Принцип One Harness

Один силовой хаб (DCC). Нет промежуточных реле-блоков. Силовые линии от DCC к потребителям.

```
АКБ+
  │
[Главный контактор] ← внешняя кнопка массы (регламент)
  │
 DCC Power In
  │
  ├── Ch01–02  60A   EHPS, резерв
  ├── Ch03–06  30A   Fan1, Fan2, Water Pump, Fuel Pump
  ├── Ch07–14  15A   ECU, Lights, Ignition coils*, салон
  ├── Ch15–22  5A    Габариты, печка, дворники, резерв
  └── HB01–02  H-bridge  Стеклоподъёмники
```

\*Если питаются отдельно от DCC — по конфигу E30.

### 5.2 Нагрузки и приоритеты

| Нагрузка | Ток раб. | Пуск | Приоритет | Тип выхода |
|----------|----------|------|-----------|------------|
| EHPS | 40–60 A | до 100 A | Критичный | 60A HS |
| Электропомпа | 8–20 A | 20–30 A | Критичный | 30A HS/PWM |
| Вентилятор ×2 | 20–30 A | 50–70 A | Высокий | 30A HS/PWM |
| Бензонасос | 8–15 A | 20 A | Критичный | 30A HS |
| ECU | 3–5 A | 5 A | Критичный | 15A HS |
| Свет ближн./дальн. | 10–15 A | — | Средний | 15A HS |
| Габариты | 3–5 A | — | Низкий | 5A HS |
| Салон (печка, дворники) | 5–15 A | — | Средний | 15A/5A |
| Стеклоподъёмники | — | — | Низкий | H-bridge |

Суммарный ток системы может достигать 130–170 A — **не коммутировать всё через одну дорожку**.

### 5.3 Каналы Power Board (аппаратная ёмкость)

| ID | Класс | Макс. ток | Кол-во |
|----|-------|-----------|--------|
| HS60 | High | 60 A | 2 |
| HS30 | Medium | 30 A | 4 |
| HS15 | Standard | 15 A | 8 |
| HS05 | Low | 5 A | 8 |
| HB | H-bridge | — | 2 |

**22 управляемых канала** + 20% логический запас в конфиге (не все назначены в E30).

Назначение функций — **только в конфигурации**, не в прошивке. См. [005](005_Configuration_Model.md).

## 6. VCM — режимы автомобиля

```
OFF → MASTER_ON → IGNITION → PRIME → READY → ENGINE_RUN → RACE → COOL_DOWN → OFF
```

| Режим | Описание |
|-------|----------|
| OFF | Масса снята / контактор открыт |
| MASTER_ON | Контактор замкнут, система жива |
| IGNITION | Зажигание, подготовка |
| PRIME | Прогрев бензонасоса |
| READY | Готов к запуску |
| ENGINE_RUN | Двигатель работает |
| RACE | Режим трассы (упрощённая телеметрия UI) |
| COOL_DOWN | Охлаждение после остановки |

Таблица «выход × режим» хранится в конфиге (`modes` в YAML).

## 7. Button Box

- STM32 + CAN FD
- События: `press`, `release`, `long_press`, `hold`, `rotate`
- Debounce и разбор клавиатуры — **внутри Button Box**
- DCC только исполняет привязки `inputs` → `rules` / `modes`
- Физическое: свет, дворники, печка, режимы, энкодеры

## 8. ECU (граница)

Только двигатель. Взаимодействие: `cooling_request`, телеметрия по CAN.  
Подробнее: [003_ECU_Architecture.md](003_ECU_Architecture.md).

## 9. Безопасность (гоночная машина)

| Механизм | Реализация |
|----------|------------|
| Fail-safe OFF | Все выходы OFF при сбое MCU / отсутствии ENABLE |
| Kill switch | Независимый аппаратный вход + разрыв контактора массы |
| Масса | Физический контактор, **не** через MCU |
| Watchdog | TPS3430 (аппаратный) + firmware WDT |
| Деградация | Отказ одного канала ≠ отказ всей системы |
| MCU завис | Выходы OFF, силовые ключи отключены |

```text
MCU завис → ENABLE снят → все каналы OFF
```

## 10. Универсальный периферийный порт (DTM12)

Все малые CAN-устройства подключаются одинаково:

| Pin | Сигнал |
|-----|--------|
| 1 | +12V (через предохранитель) |
| 2 | GND |
| 3 | CAN_H |
| 4 | CAN_L |
| 5 | IGN (ключ зажигания, опц.) |
| 6 | IN1 |
| 7 | IN2 |
| 8 | OUT1 (low power, опц.) |
| 9 | OUT2 |
| 10 | WAKE |
| 11 | Shield |
| 12 | Reserved |

Button Box, будущий Dash, телеметрия — один стандарт разъёма.

## 11. Разъёмы (классы)

| Класс | Тип | Назначение |
|-------|-----|------------|
| Логика / CAN | Deutsch DTM | Button Box, периферия |
| Средний ток | Deutsch DT | Салон, средние нагрузки |
| Силовой | Deutsch DTP | EHPS, вентиляторы, насосы |

> Один тип разъёма = один класс сигналов. Доступность: AliExpress / LCSC.

## 12. Входы DCC (Logic Board)

| Тип | Кол-во | Назначение |
|-----|--------|------------|
| Аналог 0–5 V | 4 | Датчики (опц.) |
| Цифровой | 6 | Масса, kill, стартер, резерв |
| Частотный | 2 | Скорость, тахо (опц.) |
| Expansion | 1 | SPI/I2C/UART — IMU, GNSS |

## 13. DevKit → DCC

```
DriveCore DevKit  →  (убрать лишнее, добавить силовую ёмкость)  →  DCC Gen1
```

DevKit отрабатывает протокол, bootloader, конфиг, Web UI до автомобильной платы.

## 14. Три принципа платформы

1. **Software Defined Vehicle** — логика в конфиге, не в пайке.
2. **One Harness** — один хаб, минимум жгутов.
3. **Field Serviceable** — замена модуля в паддоке за 5–10 мин, автообнаружение по CAN.
