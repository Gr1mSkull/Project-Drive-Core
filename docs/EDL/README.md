## EDL-001: STM32G474 вместо H7

**Дата:** 2026-07-07  
**Статус:** accepted

### Контекст
Выбор MCU для Logic Board DCC.

### Решение
STM32G474RET6 — достаточно для PDM+VCM+CAN+диагностики.

### Альтернативы
STM32H5/H7 — избыточны для Gen1 (дисплей, Ethernet, файловая система не нужны на MCU).

### Последствия
Меньше стоимость и сложность; ESP32 берёт на себя сеть и Web UI.

---

## EDL-002: Отдельная Radio Board (ESP32)

**Дата:** 2026-07-07  
**Статус:** accepted

### Контекст
Как подключать планшет и делать OTA.

### Решение
ESP32-S3/C6 на отдельной плате внутри DCC. STM32 — автомобиль, ESP32 — Wi-Fi/BLE/Web/OTA.

### Альтернативы
ESP32 как главный MCU — хуже real-time. USB к планшету — ненадёжно в вибрациях.

### Последствия
Модуль можно не ставить или заменить без переделки Logic Board.

---

## EDL-003: Smart High Side вместо дискретных MOSFET

**Дата:** 2026-07-07  
**Статус:** accepted

### Контекст
Архитектура силовых выходов Power Board.

### Решение
Infineon PROFET+ / BTS — встроенная защита и диагностика.

### Альтернативы
MOSFET + gate driver — дешевле, но сложнее защита и PCB.

### Последствия
~40–50% BOM — силовые ключи; выше надёжность и диагностируемость.

## EDL-004: Web UI вместо нативного приложения (Gen1)

**Дата:** 2026-07-07  
**Статус:** accepted

### Контекст
HMI для диагностики и настройки.

### Решение
Встроенный Web UI на ESP32; планшет Android/iPad как браузер.

### Альтернативы
Собственный Dash — месяцы разработки. Нативное приложение — позже (Gen2/3).

### Последствия
DCC = сервер, планшет = клиент. Dash исключён из Gen1.

## EDL-005: ECU не управляет кузовными нагрузками

**Дата:** 2026-07-07  
**Статус:** accepted

### Контекст
Разделение ответственности DCC vs ECU.

### Решение
ECU — только двигатель. Вентиляторы, помпа, свет, масса — DCC.

### Альтернативы
ECU коммутирует вентиляторы — дублирование и путаница проводки.

### Последствия
Чёткая граница модулей, проще диагностика.

---

## EDL-006: VCM интегрирован в DCC

**Дата:** 2026-07-07  
**Статус:** accepted

### Контекст
Отдельный VCM vs объединение с PDM.

### Решение
VCM внутри DCC для E30 — одна точка логики режимов.

### Альтернативы
Отдельный VCM — оправдан только при десятках ECU (GT3/LMP уровень).

### Последствия
DCC переименован из «PDM»; это Vehicle Power Controller.

## EDL-007: Трёхплатная архитектура DCC

**Дата:** 2026-07-07  
**Статус:** accepted

### Контекст
Размещение 22+ силовых каналов.

### Решение
Logic Board + Power Board + Radio Board в одном корпусе.

### Альтернативы
Всё на одной плате — сложная разводка, дорогой ремонт силовой части.

### Последствия
Gen2 меняет только Power Board; DevKit → DCC по той же логике.

---

## EDL-008: Одна шина CAN FD в Gen1

**Дата:** 2026-07-08  
**Статус:** accepted

### Контекст
Два CAN (ECU + Body) vs одна общая шина.

### Решение
**Одна** линейная CAN FD: DCC — ECU — Button Box.

### Альтернативы
Две шины — избыточно для 3 узлов; усложняет жгут.

### Последствия
Второй трансивер на Logic Board — задел Gen2. См. `docs/004`, `docs/001`.

## EDL-009: DriveCore Protocol (именованные сообщения)

**Дата:** 2026-07-08  
**Статус:** accepted

### Контекст
Избежать «магических» CAN ID без документации.

### Решение
DCP v0.1: CAN ID = f(MessageClass, DeviceType, Instance); payload с `proto_ver` + `msg_type`.

### Альтернативы
Произвольные ID — не масштабируется на ABS, телеметрию и т.д.

### Последствия
Единый стандарт для всех модулей; позже codegen из `.dcmsg`.

## EDL-010: Бинарный SPI, JSON только для браузера

**Дата:** 2026-07-08  
**Статус:** accepted

### Контекст
Обмен STM32 ↔ ESP32.

### Решение
DCPI: бинарные кадры + CRC. JSON только REST/WebSocket для планшета.

### Альтернативы
JSON на SPI — лишняя нагрузка на STM32.

### Последствия
`STATE_PUSH` 50 ms; ESP32 агрегирует JSON для UI.
